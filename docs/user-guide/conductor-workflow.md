# Conductor Workflow 原理与实战指南

> **把项目上下文当做最高级别的代码资产** —— 这是 Conductor 流派的核心主张。

长久以来，当我们面对大语言模型（Agent）指挥一个中大型项目时，最让人头疼的就是**上下文遗忘与自说自话**。你告诉它不要用 Class，它过几轮对话又给你生成了一堆 Class；你让它只写某个函数的单元测试，它顺手把你的主业务代码给“重构”崩了。

为了彻底解决 AI “记吃不记打”的游击战式开发，Meshy 引入了这套 **Conductor Workflow（轨道指挥官工作流）**。通过明确的四个阶段与严明的纪律控制，将混沌的对话流转化为可被审计的工厂流水线。

---

## 一、四大纪律基石 (The 4 Lifecycle Skills)

Conductor Workflow 由四个层层递进的专属 Skill 组成。只有你遵循正确的起手式与轨道铺设，才能在最后收获高质量、毫无偏离的代码验收。

### 1. 筑基 🧱 (`+conductor-setup`)
这是你面对新项目、新架构（甚至是从零起步）的**唯一**起手动作。
- **目标**：不写一行代码，只建立全局宪法。
- **机制**：呼唤此技能后，Agent 会停止一切行动，化身架构师对你进行**采访**。它会逼问你产品的终极目标、技术选型红线、以及流派规范。最终，它会自动在你的工作区 `.meshy/context/` 目录下生成 `product.md`, `tech-stack.md`, `workflow.md`。
- **效果**：从此以后，任何新加入的 Agent（包括未来的你重启客户端），都将无条件继承并遵循这三份宪法，彻底根治大模型的“幻觉与跑偏”。

### 2. 立项 📐 (`+conductor-track`)
当你准备开发一个新功能（如“登录系统重构”或“暗黑模式支持”）时使用。
- **目标**：将需求翻译为无歧义的执行清单。
- **机制**：你抛出粗糙的想法，Agent 会基于你之前的 Context 进行审查，指出漏洞并与你确认边界（Gap Analysis）。随后，它会在 `.meshy/plans/<track-id>/` 目录下生成规范的 `spec.md`（需求约定）与 `plan.md`（原子级的打勾任务树并附带 Phase 测试卡点）。
- **效果**：大模型替你把一团乱麻的思路，拆解为了连实习生都能照着做的流水线工单。

### 3. 行军 🔨 (`+conductor-implement <track-id>`)
一旦计划敲定，这是真正的苦力活指令。
- **目标**：严酷的纪律执行者，只认 `plan.md`，不认其他。
- **机制**：你可以任意启动终端运行 `npm run dev` 放着不动（借助于新的异步 Background Terminal 功能）。Agent 将逐个摘取 `plan.md` 的 `[ ]` 任务，自动**编写代码 -> 测试绿灯 -> 打勾标记 `[x]` -> 拾取下一个**。
- **卡点防御**：如果在一个 Phase 的队尾遇到了 Checkpoint 拦截，它必定强制挂起并向你报告（如：“我已经写完了 API 层，请您使用 Postman 戳一下看通不通，确认后再搞 UI”）。

### 4. 收检 🔍 (`+conductor-review <track-id>`)
干完活后用于质量验收与安全防护。
- **目标**：防呆、防毒、防丑。
- **机制**：对照最开始的 `spec.md` 与实际文件系统的修改 diff，对整个生命周期进行极其挑剔的找茬。它会出具一份高密度的纠错单，分为 Critical（致命缺陷必须返工）与 Minor（建议改进的 Code Smell）。

## 二、实战演练：如何完成一次高质量开发？

假设你现在要在自己的前端项目里集成一套 **Stripe 订阅支付系统**。传统的做法是直接在对话框里发泄：`“帮我写一个 Stripe 支付面板，带 Webhook”`。这种野路子注定会引发后续几千行代码的连续爆红。

使用 Conductor Workflow 的正确姿势如下：

### Step 1: 确保 Context 存在
看看侧边栏或文件树，如果你的项目尚未被 Meshy 接管过，第一句话应该是：
> `+conductor-setup 我这边是一个 SaaS 模板，我们要准备接入 Stripe。请进行采访并制定架构大纲。`

*（经过问答，你的 `.meshy/context` 配置完毕：限定使用 TypeScript 强类型，禁止行内 CSS，一律使用 Tailwind，后端选定 Next.js App Router API 模式。）*

### Step 2: 建立 Stripe 轨道
> `+conductor-track 我们需要在这个项目中加入 Stripe 订阅面板，同时包含对基础版的免费限制逻辑，并暴露 Webhook 收据。`

*（大模型会指出：“关于退款逻辑需要考虑吗？”你答复“暂不考虑”。随后它会生成包含 Phase 1: 基础设施配置、Phase 2: 核心 UI 面板、Phase 3: Webhook 后端的详尽 `plan.md` 轨道。）*

### Step 3: 开始流水线冲刺
查阅生成的 `plan.md` 无误后：
> `+conductor-implement stripe-integration`

接下来就是见证奇迹的时刻。你会看着进度条（大模型的一轮轮自驱动思考）疯狂闪烁，它会按图索骥去装库、建类型文件、落端点代码。它不会跨过 Phase 1，直到你点头通过。

### Step 4: 专家会诊签单
当终端宣告 Phase 3 The End 时：
> `+conductor-review stripe-integration`

耐心等待几秒，一份极具压迫感的验收报告将呈递在你面前：可能它发现了 Webhook 验证并没有校验签名来源，或者密钥明文暴露。有了这份报告，再打回重做，你就彻底避免了上线事故。

## 三、Conductor 的进阶优势总结
- **可恢复性 (Resilience)**：假如今天下班你关机了，明天打开 Meshy 继续打 `+conductor-implement`，它会自己看 `plan.md` 之前打到哪个勾，从断点直接**光速断点续传**，永不懵逼。
- **抗上下文腐败 (Context Decay Prevention)**：以往超百轮的对话树必然导致 AI 遗忘初始设定。通过 `.meshy/context` 持久化，这是 AI 的物理级永久记忆，它总能准确认出你的项目到底是个什么底子。
- **解耦式管控**：你的身份不再是“帮 AI 改 Bug 的保底程序员”，你升级为了“审查流水线质量的工程大员”。
