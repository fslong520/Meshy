# Meshy Agent 架构设计 v4：全矩阵与超编排

> 基于对多种 Agent 框架（oh-my-opencode / Conductor / Claude Code）的深度剖析，结合不同人群的梯度需求，打造一套既开箱即用、又具备极高编排自由度的下一代 Agent 架构。

---

## 一、核心痛点与演进方向

### 1. 为什么之前的设计不够？
1. **版图不全**：缺乏对视觉分析（Multimodal）、计划审查（Metis）、代码/结果验收（Momus）这一类保障质量和覆盖边缘场景的专属角色。
2. **缺乏"反思与多选"机制**：顶尖的 AI 开发体验（如 Codex 在 Claude Code 中的表现），不是"一条路走到黑"，而是在关键分岔路口停下来思考，向用户提供多个架构选择。当前的 Agent 设计没有内置这种停顿与探索的机制。
3. **协作阻力**：Agent 之间是孤岛，未能形成真正的协同工作流，缺乏靠 Skills 驱动的动态调用网络。

### 2. V4 架构的核心解法
1. **完整的专家矩阵**：提供 10 个专业化预设 Agent，覆盖从视觉读取到架构验收的完整生命周期。
2. **分人群的交互梯度**：
   - **小白/效率党**：使用 `Default / Coder` + `Smart Mode`，开箱即用。
   - **进阶/严谨党**：使用 `Planner` 构建计划 -> `Executor` 执行 -> `Reviewer` 验收。
   - **极客/专家党**：自定义组合 Agent，通过编写高阶的 **Flow Skills**（流程编排技能），让 Agent 之间互相调用，实现极限控制。
3. **反思回路与多分支决策**：在特定的 Agent 提示词和执行流程中，内置"自我辩论"和"多方案输出"的阶段。

## 二、Agent 角色全景矩阵

本次升级将核心参与者扩张为了一个十人战队。我们使用通俗易懂但高度专业的名词，以便用户一瞥即知其职能。

### 2.1 用户视角：下拉菜单可选的 Agent

| Agent | 徽章 | 前身 (OmO/Conductor) | 核心职责与特色 | 工具权限 |
|---|:---:|---|---|---|
| **Default** | 💬 | Sisyphus | **大管家**：零注入，适用于通用问答、文案翻译、简单调研。处理不需要复杂上下文的任务。 | 全面 |
| **Coder** | 💻 | (新增) | **主力工程师**：注入代码库上下文和技术栈。日常最常用的单任务写代码助手。 | 全面 |
| **DeepCoder** | 🧠 | Hephaestus | **极客探路者**：具备自主探索意志。遇到复杂系统时，它会不断阅读、做实验，不轻易交卷。 | 全面 |
| **Planner** | 📋 | Prometheus | **战略规划师**：不写代码，只通过采访模式收集需求，产出结构化的需求文档和实施计划（Phase/Task）。 | 读 + 写特定目录 |
| **Executor** | ⚡ | Conductor | **纪律执行者**：闭口不言，只看 Planner 的计划行事。严格遵循 TDD 和 Checkpoint，专注完成当前 Task。 | 全面 |
| **Advisor** | 👁️ | Oracle | **资深架构师**：只读分析，帮你做技术选型、审查架构图，给出深思熟虑的利弊权衡。 | 只读 |
| **Explorer** | 🔍 | Explore | **雷达侦察兵**：专精**本地项目**代码库的快速遍历、Grep 搜索和模式匹配。 | 本地搜索 |
| **Librarian** | 📚 | Librarian | **外部图书管理员**：专精**互联网和文档**检索。查找官方 API 文档、开源库用法。 | 外部网络通信 |
| **Reviewer** | ✅ | Momus | **铁面验收官**：验收代码库改动或执行结果。对照规范找出安全漏洞或逻辑缺陷。 | 只读 |
| **Scanner** | 🖼️ | Multimodal Looker | **视觉解析仪**：专攻多模态。丢给它设计稿截图、白板照片、PDF，它将其转换为需求文字或组件结构。 | 读取文件/图片 |

## 三、三维正交模型体系深化

我们的底座依然是 `Agent × Mode × Skill`，但在每一个维度都做了深化。

### 3.1 模式 (Mode) 定义的反思与确认

Mode 决定的是 AI 的 **自主能动性边界**，也就是它有多大权力自作主张。

| Mode | 自主能动性边界 | 适用场景 |
|---|---|---|
| **Standard** | **不采取动作。** 你问什么它答什么，偶尔给建议。没有你的显式命令，绝对不执行修改。 | 轻量级咨询，危险操作前的反复确认 |
| **Smart** | **主动探索，谨慎修改。** 它会自己调工具去看文件、查资料，但在生成大量代码或做出破坏性更改前，会向你请示。 | 日常最实用的写码抓虫模式 |
| **Auto** | **全自动拉满，直到完成。** 一旦下令，它会穷尽手段完成目标。如果中间有疑问，它自己尝试解决。 | 跑自动化测试修复、大批量文件重构 |

### 3.2 深度：自我辩论与多选机制 (Multi-Option Output)

这是吸收 Codex 执行体验精髓的关键点。在涉及到复杂设计时，单路求解往往是错的。

**触发条件**：
当使用 `DeepCoder` 或 `Advisor` 时，或者在 Planner 的需求澄清阶段，系统内置了对复杂度的感知。

**行为表现**：
系统会在 prompt 层明确规定：
*当遇到具有多种实现路径的工程难题，或者上下文中有含糊不清的业务逻辑点时，不要急于选定一条路。你的任务是暂停执行，并向用户提供一个类似于选择题的回复。*

**示例交互**：
```
我发现这里需要建立一套状态管理。方案有两层分歧，请你指示：
选项 A (Redux 派系): 规范严谨，但样板代码多，对当前体量可能略重。
选项 B (Zustand/Jotai): 轻量灵活，能够快速满足需求。
选项 C (自定义 Hooks + Context): 零依赖，但后续维护要求高。

请问你倾向于哪种？或者输入你自己想用的其他库：[   ]
```
在这个停顿点，用户甚至可以通过自定义文本提供**选项外**的想法。

## 四、分层编排：高阶玩法的终极武器——Skills 联动

真正的威力在于不要把所有流程写死，而是把这些高阶工作流做成能自由组合的 **Skills**。

### 4.1 Skill 层级划分

我们将 Skill 分为两大类：**工具组 (Tool Skills)** 与 **流程编排组 (Flow Skills)**。

#### 1. Tool Skills（带工具与提示语的增强包）
- `+git-master`：赋予原生 Git 的熟练掌控力。
- `+playwright`：赋予浏览器自动化测试能力。
- `+deep-research`：长耗时深入网络调研的技术规范。

#### 2. Flow Skills（工作流剧本，专为联动 Agent 打造）
这些是高阶用户能将产品活力释放出来的关键。它们本身不只是系统 prompt 的补丁，更像是一份**指挥手册**。

- **`+strict-tdd` (严格测试驱动)**
  - 教导当前的执剑者（如 `Executor` 或 `Coder`）如何先写测试，再写实现。
- **`+metis-analysis` (漏洞嗅探)**
  - 这是对 OmO 中 Metis 缺失的补全。当加载这个技能在 Planner 身上时，Planner 产出初始计划后，会触发一次**内部反思回合**（自己作为 Metis 去挑刺），然后才输出最终的 Plan。
- **`+momus-review` (苛刻审查)**
  - 配合 `Reviewer` Agent 使用，大幅上调审查标准，涵盖安全漏洞、性能开销审查。
- **`+conductor-lifecycle` (对标 Conductor 全流程管理)**
  - 加载这个技能后，系统将强制执行严苛的项目管理规范。
  - 要求建立 `tech-stack.md`、`product.md`。
  - 要求所有的实现必须遵循 Phase Checkpoint 和人工验证签字，否则不许进行下一个任务。

### 4.2 Agent 间互相调用的化学反应

怎么实现互相调用？
在更高阶的设计中，我们可以为超级应用场景赋予主 Agent（如 `Default`）使用 `/call <Agent_Name>` 子命令或特殊 Tool 的权限。而在现阶段，我们依赖**知识载体接力**和 **Flow Skills**。

#### 示例工作流：一篇新功能的落地
**目标人群：严谨党/团队协作**

1. **构思阶段**：用户向 `@Scanner` (Multimodal) 发送了一张 Figma 截图，并要求生成 UI 组件的需求描述。
2. **规划阶段**：用户选定 `@Planner`，并挂载 `+metis-analysis` 技能。
   - `Planner` 读取刚才生成的描述，开始提问。
   - 在成文前，触发 `metis-analysis` 的漏洞嗅探思考（"等一下，这个弹窗的关闭逻辑没有明确，如果不点 X 点背景能关吗？"），补充漏洞。
   - 最终将结果写入 `.meshy/plans/feature-x.md`。
3. **执行阶段**：用户切换到 `@Executor`，挂载 `+strict-tdd` 技能，输入命令 `/start-work feature-x`。
   - `Executor` 在后台静默执行，直到一个子阶段通过了测试，抛出一个 Checkpoint 等待验收。
4. **验证阶段**：用户发现某个复杂状态有问题，求助于 `@Advisor`。
   - `Advisor` 进行多分支思考分析，给出几个可能的性能妥协方案供用户选择。

这种"拼积木"的方式，把极具魅力的流程通过 Skill 释放了出来，绝不捆绑普通用户的日常简单使用。

## 五、统一的 Command 体系规范设计

Commands (/ 斜杠指令) 本质上是**快捷启动宏**，它们在特定的时候会**隐含强行绑定/调用某个 Agent**。

| 命令 | 用途 | 幕后关联与触发 |
|---|---|---|
| `/init` | 建立项目上下文 | 调用 Default，扫描结构，创建 `.meshy/context` 目录 |
| `/plan <desc>` | 快捷新建并启动规划流 | 自动切至 `Planner` 开始采访，产出 `.meshy/plans` 下文档 |
| `/start-work` | 按现有计划序列化执行 | 自动切至 `Executor`，寻找最新的 plan 推进 |
| `/review` | 针对某文件或最近 commits 的审查 | 自动切至 `Reviewer` 产出报告 |
| `/search` | 快速本地检索 | 自动切至 `Explorer` |
| `/research` | 快速网络研读 | 自动切至 `Librarian` |
| `/doctor` | 即时错误根因诊断 | 唤起 `Advisor` 扫描当前环境、进程日志与 recent changes |
| `/compact` | 沉淀上下文历史 | (系统层) 将冗长记录提纯，减轻后续 Agent 的记忆压力 |

## 六、总结对比与实施指南

### 相比 v3 的关键升维
1. **完善了专家版图**：不仅将检索精细拆分为了探查本地的 `Explorer` 与探查广域知识的 `Librarian`，还补全了多模态专家 `Scanner`。
2. **去硬编码，拥抱 Skills 剧本化**：那些听上去很美好的严苛审查流程（Metis/Momus），不再是不可见的底层黑盒。我们将它们抽离成了 `+metis-analysis`, `+momus-review`, `+strict-tdd` 等一系列 **Flow Skills**。用户可以选择性挂载，组合出千变万化的开发风格。这赋予了高玩群体重构系统表现的能力。
3. **内置智慧停顿（自我辩论）**：明确在 `DeepCoder` 等深度执行角色中内置提示词，教导其在面对极高复杂度和歧义时，不盲动，向用户甩出选项卡进行沟通。

此等分层架构，既确保了小白通过简单的 `@Coder` + `Smart Mode` 能够直接体验到高效辅助开发，也通过 `Agent + Flow Skills` 的无限组合解开了极客们的创造力枷锁，真正实现了产品生命力与拓展弹性的最大化。
